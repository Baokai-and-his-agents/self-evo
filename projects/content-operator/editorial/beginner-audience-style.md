# 普通读者内容风格卡

## 写给谁

聪明、忙碌、没写过代码，但正在判断 AI 是否值得进入自己工作和生活的人。

## 每篇必须做到

1. 从熟悉场景进入，不从术语定义进入；
2. 每篇只有一个主要观点；
3. 术语首次出现时用日常比喻解释；
4. 至少给一个可以直接尝试的低风险动作；
5. 同时写清“能做什么”和“不能做什么”；
6. 事实、建议和假设分开；
7. 不要求读者了解 self-evo、GitHub、评测框架或编程。

## 避免

- “颠覆”“淘汰所有人”“百分百自动”“零风险”；
- 没有来源的效率、增长或失败率数字；
- 连续堆叠 framework、workflow、orchestration 等英文词；
- 把 AI 写成拥有责任和后果意识的人；
- 为了显得通俗而牺牲边界准确性；
- 用焦虑代替真实价值。

## 推荐结构

```text
熟悉场景
→ 一个意外冲突
→ 用日常语言解释核心概念
→ 三到五个具体判断
→ 风险边界
→ 一个今天就能做的小实验
```

## self-evo 的使用方式

self-evo 是幕后证据库，不是默认主题。只有当真实经历能让普通读者更容易理解问题时，
才用一句到一段案例说明；正文不得要求读者先理解项目结构。
