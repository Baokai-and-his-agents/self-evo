# AI Agent 内容方法论 v1

## 1. 样本与方法

- 调研日期：2026-07-10
- 样本数量：25
- 研究问题：高可信 AI Agent 实践内容如何选择问题、组织证据、呈现失败与形成可执行结论？
- 纳入标准：公开可读、作者可识别、包含具体经验/模式/限制或有价值反例
- 排除标准：登录墙、无正文、纯聚合、来源不明数字、只讲能力不讲证据
- 主要限制：样本偏工程团队和厂商；没有统一的流量、转化与受众数据

完整记录见 [`source-log-2026-07-10.md`](source-log-2026-07-10.md)。

## 2. 执行摘要

高可信的 Agent 内容不是“信息最多”，而是围绕一个真实问题，公开约束、选择、证据、
失败和结果。样本中最稳定的结构是：为什么需要解决 → 为什么常见方案不够 → 采取
什么边界明确的方法 → 如何验证 → 哪里仍然失败 → 读者下一步能做什么。

对 content-operator 最重要的差异化资产不是追逐新闻，而是 self-evo 已留下的 Issue、
claim、PR、测试、失败修正和人工决策记录。第一批文章应优先讲这些别人难以复制的
真实证据。

增长效果目前证据不足。本方法论只对内容可信度、可复查性和表达结构提出建议，不把
它们宣称为阅读量或涨粉保证。

## 3. 目标受众与真实需求

| 受众 | 需要解决的问题 | 可提供的真实证据 | 证据强度 |
|---|---|---|---|
| 正在搭 Agent 的开发者 | 如何从 demo 走到可靠运行 | self-evo 的边界、测试、claim、PR 与运行记录 | 高 |
| 想用 Agent 改造工作的个人/团队 | 哪些判断能交给 Agent，哪些必须保留给人 | Issue #40/#42 的决策点和权限分工 | 高 |
| 关注 AI 实践但不写代码的读者 | Agent 项目真实会在哪里失败 | 可用故事化方式解释测试盲区、身份和治理问题 | 中高 |
| 寻找“全自动增长方案”的读者 | 是否存在无需判断的自动内容机器 | 当前证据支持否定边界，不支持增长承诺 | 中 |

## 4. 有效内容模式

### 模式 A：单一真实问题 + 明确约束

- 现象：高质量文章通常只解决一个问题，例如上下文、评测、暂停恢复或工具设计。
- 为什么有效：读者能判断适用性，作者也能提供足够深的证据。
- 证据：S02–S04、S11、S17。
- 反例：宽泛“Agent 完全指南”容易成为名词列表。
- 建议：每篇 self-evo 内容只承诺回答一个具体问题。

### 模式 B：How we built + 失败与取舍

- 现象：生产复盘同时解释架构、选择原因、失败、修正和剩余限制。
- 为什么有效：过程证据比最终能力声明更可信。
- 证据：S05、S06、S09、S18。
- 反例：只展示成功 demo 会隐藏生产差距。
- 建议：文章必须包含“哪里没按计划工作”。

### 模式 C：模式/层级 + 何时不用

- 现象：强方法文章不仅给方案，还给选择条件和不用该方案的情况。
- 为什么有效：读者获得决策工具，而不是复制模板。
- 证据：S01、S16、S19、S21。
- 建议：每篇文章加入“什么时候不适用”。

### 模式 D：证据链而非单一结果

- 现象：可靠性文章把 trace、dataset、metric、quality gate 和人工判断连接起来。
- 为什么有效：Agent 是多步且非确定的，单一成功截图不能代表可靠性。
- 证据：S02、S06–S08、S17、S20。
- 建议：优先展示 issue → diff → test → review 的完整链路。

### 模式 E：反直觉标题 + 可验证框架

- 现象：“何时完全不用 Agent”“为什么 demo 会失败”能建立张力。
- 为什么有效：挑战流行假设，但必须给出判断框架和真实证据。
- 证据：S19、S23–S25。
- 风险：反直觉标题最容易滑向夸张和无来源失败率。
- 建议：标题可以有冲突，正文证据不能降级。

## 5. 选题原则

优先级：

1. 我们拥有真实、别人难以复制的证据；
2. 解决一个明确问题；
3. 能展示失败、修正或取舍；
4. 读者看完能做一个具体决定或动作；
5. 核心结论不依赖无法核验的平台数据。

应避免：

- 新闻摘要和工具罗列；
- “十大最佳 Agent”式无透明标准排行；
- 把官方能力描述改写成自己的实践经验；
- 没有分母和来源的成功率/失败率；
- 把单个案例包装成普遍增长规律。

## 6. 标题与开篇

可复用标题结构：

- `我们如何……以及真正困难的部分`
- `为什么……不是正确的第一步`
- `X 次真实运行后，我保留/砍掉了什么`
- `当所有测试都通过，功能为什么仍然失败`
- `什么时候应该使用 workflow，而不是 Agent`

开篇应在前三段给出：真实场景、意外冲突、本文拥有的证据。禁止用“颠覆”“彻底
改变”“百分百自动”等无法兑现的表达。

## 7. 正文结构

推荐固定骨架：

1. **问题：** 真实任务和为什么值得关心；
2. **约束：** 权限、成本、数据、时间和不可做事项；
3. **原方案：** 最初怎么想，为什么不够；
4. **执行：** 关键步骤和决策点；
5. **证据：** Issue、diff、测试、日志、来源或结果；
6. **失败：** 哪些地方出错，单元测试为什么没发现；
7. **修正：** 做了什么以及代价；
8. **边界：** 什么时候不适用、仍未知什么；
9. **下一步：** 读者可以执行的一个动作。

## 8. 平台适配假设

- 微信公众号：适合完整复盘、图示和证据链；
- Twitter/X：适合把一个冲突拆成短线程，链接回完整证据；
- 同一素材先写完整事实版本，再压缩，不反向用短观点拼成长文。

这些是待验证假设，不是增长结论。前五篇发布后应记录编辑耗时、完成率、有效回复、
收藏/转发等可获得数据，再判断平台策略。

## 9. 失败模式

| 失败模式 | 识别信号 | 避免方式 | 证据 |
|---|---|---|---|
| 框架先于问题 | 标题先出现工具名，正文没有真实任务 | 先写问题、约束和不用 Agent 的条件 | S01、S19、S21 |
| Demo 等于生产 | 只有一次成功截图 | 展示多次运行、失败与质量门 | S02、S06、S18、S23 |
| 厂商结论无交叉验证 | 数字指向自家产品且无原始出处 | 降低证据等级，寻找官方/原始来源 | S08–S10、S23 |
| 清单没有决策条件 | 每项都“很重要” | 增加适用条件、反例和排序理由 | S04、S16 |
| 把推断写成事实 | 使用“证明增长”“一定有效” | 明确事实、推断、假设与未知 | 全部样本审查 |
| 自动化越过人类责任 | 没有终审和停止点 | 写清触发条件、责任人和人工决定 | S10、S21 |

## 10. 结论可信度

高置信：

- 真实问题、过程证据、失败和边界能提升技术内容的可复查性；
- 单问题深挖、How-we-built、决策框架是本样本中反复出现的强结构；
- self-evo 的真实审计资产比泛化 AI 新闻更有差异化。

中置信假设：

- 反直觉标题可能更容易吸引目标读者，但必须保持证据一致；
- 长文与短线程分工可能适合公众号/X 的组合。

证据不足：

- 哪种标题带来更高点击率；
- 最佳发布频率和发布时间；
- 公众号与 X 的增长路径；
- 20–30 个工程样本是否代表更广泛中文读者偏好。

下一轮用前五篇真实运行数据验证，而不是继续扩大规则或追加二手榜单。
