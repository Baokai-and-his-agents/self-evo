## Agent 推荐的前三项

### 推荐 1：当 221 个测试全部通过，真实功能为什么仍然失败

- 推荐理由：冲突强、证据完整，能展示“单元测试通过不等于真实闭环成功”的普遍问题。
- 关键证据：PR #37 run summary、四轮复核、HN/arXiv 真实端到端修正；外部对照 S02、S06、S20。
- 预计工作量：中。
- 最大风险：需要逐项核对真实 bug，不能把辅助 ignore/schema 测试与 Runner bug 混为一谈。
- 建议角度：从“测试报告看起来完美”开场，解释为什么真实输入、环境和端到端证据不可替代。

### 推荐 2：为什么我禁止 Agent 修改自己的规则

- 推荐理由：self-evo 最独特且容易理解的设计选择，兼具安全、治理和实际工作流。
- 关键证据：`rules/**`/`data/**` 边界、rule proposal、CODEOWNERS、Issue/PR 审计链；外部对照 S01、S10、S21。
- 预计工作量：中低。
- 最大风险：不能把“不能改规则”说成“不能学习”，需要解释记忆和 proposal 路径。
- 建议角度：把 Agent 比作能提出制度修改建议、但不能自行批准制度的人。

### 推荐 3：“全自动 Agent”为什么仍然必须让人做选择题

- 推荐理由：能连接内容生产与 Agent 系统设计，受众最广，也直接回应“自动化到底到哪里”。
- 关键证据：Issue #40/#42 的人工决策点、发布禁止、claim/merge 身份分工；外部对照 S10、S19、S21。
- 预计工作量：中。
- 最大风险：观点容易泛化，必须写出具体触发点和责任承担。
- 建议角度：全自动不是无人批准，而是系统在需要批准前已经把选择压缩到三个高质量选项。
