# Content Operator Loop —— 内容资产冷启动的方向校准与任务定义

> 本文档记录一次方向校准讨论的完整结论，并据此定义给执行 Agent 的第一个真实任务。
> 文档自包含：只读这一份即可理解背景、结论与待办，无需回溯聊天上下文。
> 日期：2026-07-08。撰写者：人类（Baokai）+ 协作 Agent（ZCode）。

---

## 0. 这份文档为什么存在

`self-evo` 仓库过去几个月积累了一套相当完整的治理与协议（`rules/`、hooks、validator、claim/lease、双身份、Draft PR 流程、Scout 来源 schema）。但读完整个仓库后，我们得出了一个不舒服但重要的判断：

> **当前仓库约 90% 是治理与协议，真正能自动产出东西的能力接近 0%。**

README 里也已经写明了最关键的一句——当前最有价值的下一步**不是增加更多框架，而是批准并实现 Scout Runner 权限方案**。换句话说，问题不在于没看见，而在于一直没动手。

这份文档的目的是：**把一个错误的指令（"重构"）替换成一个正确的指令（"跑通一条最小但完整的内容生产闭环"）**，并给出明天的强模型可以照着执行的具体边界与验收标准。

---

## 1. 讨论：为什么"重构"是错误指令

强大的模型最擅长的事情之一，是把已有材料重排、抽象、分层、泛化。而"重构 self-evo"这个指令会几乎必然导向：

- 把 `rules/` 拆得更细更优雅；
- 给 Scout 设计更多抽象基类与接口；
- 引入新的中间层；
- 把单 Agent 流程"优化"成看起来更漂亮的版本。

每一件都很专业，但每一件都让项目离"自动产出视频/文章/策略"更远。**这是在为还没发生的复杂度建制度**，违反项目自己的第 6 条原则"复杂性由证据触发"。

正确方向不是重构，而是**用这个强模型把一条最小但完整的产出闭环真正跑起来**。

---

## 2. 目标链：从最底层往上推导

这是整次讨论最重要的产物。过去几轮一直在围绕一个**想当然的中间指标（流量）**优化，直到最后才挖出真实目标。梳理后，目标链如下：

```
最底层目标: 让别人（朋友圈 / 他人）认为"我会用 AI 解决问题"
        ↑ 证据是
中层产出:  self-evo 这套系统真的能自动运营内容
        ↑ 展示在
两个窗口:  公众号（品牌锚点） + Twitter（学习沙盒）
```

每一层为上一层服务。想通这条链，就知道哪些事重要、哪些不重要。

### 2.1 关键转折：真实目标不是流量

最初几轮讨论，默认目标是"做一个高流量的公众号"。但人类最终澄清了真实意图：

- **公众号**：为了在朋友圈建立"AI 能力"的人设，让朋友遇到 AI 需求时想起人类。**不直接盈利。**
- **Twitter**：纯粹是试验场，对流量没有需求。**不直接盈利。**
- **共同目的**：建立一个给其他人看的"能力展示窗口"。

结论：这三个目标里，**没有一个需要高流量**。之前围绕"涨粉、阅读量、agent 从反馈学规律"的一整套逻辑，全部基于"需要流量增长"这个未被验证的前提。前提没了，结论要重来。

### 2.2 公众号的真实任务重新定义

如果目标是流量号，需要击败算法、获取自然流量、优化点击率。但如果目标是"让朋友记住人类、觉得人类懂行"，这是完全不同的命题。目标受众不是几万个陌生人，而是几十个具体的人——**他们已经认识人类，需要的是在他们心智里占据"AI"这个标签**。

这件事的关键不是阅读量，而是三件事：

1. 他们真的读到了内容（不是靠系统推荐，是人类主动出现在他们的视野里）；
2. 他们读完后觉得"这家伙懂行"（感知权威性，perceived authority）；
3. 重复出现，形成记忆（一致性比单次爆款重要 10 倍）。

这与"做流量号"是完全不同的工作方式：流量号要**迎合算法**，这里要**植入心智**。做法、节奏、成功标准都不一样。

### 2.3 Twitter 的角色重新定义——从"第二个战场"变成"沙盒"

上一轮判断曾建议别同时做两个平台。但在明确 Twitter"不要求结果、纯试验"后，它反而成了整个方案里最有价值的一块：

**Twitter 是天然的 Sandbox（学习沙盒）。**

- agent 可以在上面随便试、随便错、随便学；
- 烧不掉任何资产——因为它本来就不指望什么；
- 它承担"agent 从反馈里学习"这个高风险任务；
- 公众号承担"对外展示稳定质量"这个低容错任务。

**高风险去低容错的地方，正好。** 这个分工比之前任何方案都合理。

| 平台 | 真实角色 | 成功标准 | 失败代价 |
|------|---------|---------|---------|
| 公众号 | **品牌锚点**——给朋友圈看的窗口 | 朋友是否记住、是否觉得懂行 | 损害专业形象 |
| Twitter | **学习沙盒**——给 Agent 试错的场地 | Agent 是否真的在迭代、变好 | 几乎为零 |

### 2.4 最重要的一层：真实产品是什么

两个平台都是"能力展示窗口"，但这句是整个讨论里最关键的认知：

> **真实产品根本不是公众号文章或 Twitter 推文，而是"我能用 AI Agent 自动化地运营内容资产"这件事本身。**

文章和推文只是这个能力的**证据**。当别人看到持续产出 AI 内容，真正被打动的不是"这篇文章写得好"，而是"这家伙居然能用 AI 把这事跑起来"。这是区别于所有其他 AI 博主的定位——**别人在展示 AI 知识，这里在展示 AI 应用能力**。

而且这恰恰是 self-evo 项目的真正价值出口：**内容不是 self-evo 的产出，内容是 self-evo 的广告。**

## 3. 讨论：为什么不能让 Agent 直接替代人类做判断

过程中出现过一种诱惑："我不太信任自己的判断力，不如让 Agent 替我做判断（替我审批、替我决定发什么）。"这个想法被反复反驳，理由如下，记录在此防止反复。

### 3.1 "Agent 判断力更好"混淆了两件事

- **调研力**：遍历信息、抓取来源、整理选项。这个 Agent 确实比人强。
- **判断力**：在选项里选一个，**并为选错承担后果**。这个 Agent 永远不会"更好"，因为它根本不在这条赛道上。

**Agent 没有任何 stake。** 它推荐错的策略，亏的不是它的钱；它写错的稿子，丢的不是它的声誉。一个不承担后果的实体，无法真正判断——它只能**推荐**。推荐和判断之间隔着一条鸿沟，叫"后果"。

### 3.2 真正的代价不是"判断差"，是"失去改进回路"

把判断权交给 Agent 后，系统会变成闭合回路，无法归因：

- Agent 出稿，没看就发 → 烂。谁的错？不知道。
- 策略回测好、实盘亏 → 是策略问题还是方向没给对？不可归因。
- 不可归因 = 没有信号 = 不知道下一步该调什么。

**判断力不是天赋，是肌肉。** 它由"判断 → 承受后果 → 发现错在哪 → 下次校准"这个循环练出来。把判断让出去的那一刻，回路就断了，判断力会永远停留在今天的水平。

### 3.3 正确的分工：Agent 补信息和选项，不替最后一票

合理的分工是：

- **Agent 给信息和候选**：遍历来源、做对比、给出 A/B/C 三个选项 + 各自理由和风险。这是它强的地方。
- **人类做最后一票**：在 Agent 给的选项里选。哪怕选错，也比"完全不选"有价值——下次知道是 A 错了，信号清晰。

这样人类不需要"判断力好"，只需要：
1. 知道自己要什么（方向、风险偏好、什么叫够好）；
2. 能从 Agent 给的选项里识别（不是从零判断，是从 3 个里挑 1 个）。

挑 3 选 1 比从零创造容易十倍。**Agent 降低做判断的难度，但不替做判断。**

### 3.4 真正的"全自动"，定义是什么

> Agent 的判断**在人类不在场时，仍然和人类会在场时一致**。

不是"没有人点 approve"。前者是工程问题，后者只是把按钮藏起来。逐步过渡到全自动，正确机制不是"让 Agent 做更多步骤"，而是"让 Agent 的决定逐步收敛到人类的决定，然后按类别放行"。但这要求先有足够多的人类判断样本作为校准基准——在此之前，Draft PR + 人工 approve 的流程不能拆。

## 4. 讨论：为什么"用公众号当试验田"会失败

过程中出现过"用公众号做试验、让 Agent 从真实反馈里学习"的想法。这个思路方向上对（它第一次引入了真实外部信号），但有一个致命陷阱必须记录在案。

### 4.1 反馈信号噪声比极低

一篇 500 阅读的文章，可能是：
- 选题好但发布时机差；
- 选题好、写法对，但标题没吸引到点击；
- 选题差但被某个大号转了；
- 纯粹运气，赶上一个热点尾巴；
- 号本身流量基础低，数据没统计意义。

如果 Agent 直接拿"阅读量"当学习信号，它学到的会是一堆**混杂了发布时间、标题党、运气、热点**的伪规律。它会学偏，而且学得很自信——因为它有"数据"。

### 4.2 单篇数据没有统计显著性

在粉丝数突破某个门槛（经验值约 500–1000，具体看领域）之前，每篇文章的阅读量主要由非内容因素决定（发布时间、当天热点、初始曝光池的随机性）。在这个阶段做的所有"反馈 → 调整"，调整的都是**噪声**。Agent 会从十几个数据点里过拟合出一套自洽但无效的"规律"。

### 4.3 现状：新号、3 粉丝、长期未更新

> **公众号的推荐算法会根据历史文章数据，决定新文章能拿到多少自然流量。**

一个 0 阅读、0 转发的号会被系统判定为"低权重号"，下一篇文章获得的自然曝光更低。这是个负反馈循环。用这种号做实验，Agent 学到的规律全是被扭曲数据污染过的——"5 阅读"不代表文章差，可能只是没被足够多人看到。

**用一个坏掉的尺子量东西，然后让 Agent 从错误的测量结果里学习规律。** 这是当前最可能让项目失败的路径之一。

### 4.4 正确的学习顺序

> **在号有权重之前，不要从自己的号的数据里学规律；从别人的号的数据里学规律。**

- **阶段一（现在起，约 2–4 周）**：Agent 只研究别人，不碰人类的号。调研 AI 领域已成功的公众号，提炼可复用模式，产出"方法论 v1"。这一阶段 Agent 学的是"这个领域什么选题普遍好"，这是公开数据，信号干净。
- **阶段二（2–4 周后）**：双轨制。按调研出的成功模式稳定发布，目的是**建号权重**+验证规律，同时积累自己的数据但暂不归因。
- **阶段三（号有基础权重后）**：这时反馈才有统计意义，才开始真正的"Agent 从反馈学习"。

## 5. 给执行 Agent 的任务定义

基于以上讨论，把原本错误的指令（"重构 self-evo"）替换为下面这个正确指令。

### 5.1 总目标（一句话）

> **跑通一套"Agent 辅助冷启动内容资产"的可验证流程，用人类的公众号和 Twitter 作为两个不同性质的试验场（一个重质量、一个重学习）来证明这套流程有效。最终交付物是流程本身，加上它产出的内容作为证据。**

### 5.2 第一个垂直切片：AI 领域公众号增长方法论 v1

这是整个任务的第一块真实可交付产出，也是明天强模型应该最先做的事。

**交付物**：一份基于真实公开数据的方法论文档，覆盖 AI 领域微信公众号的增长模式。

**调研维度**（至少覆盖）：
- 选题模式（高频选题类别、选题来源、选题更新机制）
- 标题模式（结构、关键词、长度、情绪触发点）
- 内容结构（开篇、正文、结尾、配图、引用）
- 发布频率与节奏
- 发布时机（时间、星期、与热点的配合）
- 增长路径（冷启动 → 起量 → 稳定期的典型阶段）
- 变现或导流路径（用于判断哪些做法不适用于本项目）

**样本要求**：
- 至少 20–50 个 AI 领域做得好的公众号（粉丝多、阅读稳定、近 3 个月在涨）；
- 必须是真实可查的公开账号，**不许编造数据**；
- 每个样本标注其成功指标的数据来源。

**验收标准**（人类如何判断这份方法论靠不靠谱）：
1. 每条结论都有对应样本支持，能追溯到具体账号；
2. 区分了"普遍规律"与"个别案例"；
3. 明确标注哪些做法**不适用**于本项目（如追求直接变现的玩法）；
4. 落到可执行——读完知道"下一篇该写什么主题、用什么标题、什么时候发"。

**硬边界**：
- ❌ 不碰人类的公众号账号（不发文章、不改设置）；
- ❌ 不编造数据或样本；
- ❌ 不改 `rules/`、hooks、validator、双身份、Draft PR 协议——治理层本次冻结；
- ❌ 不引入新依赖/框架，除非先说明理由并等待批准；
- ✅ 遵守项目第 4 条原则"先复用，再自研"——先回答"有没有现成开源方案/研究"，再决定写不写。

### 5.3 后续切片（待 v1 验收后再启动，不要现在做）

- **切片二**：基于方法论的"选题提案器"——每次产出 3 个候选选题 + 推荐理由，人类做选择题（对应 §3.3 的分工）。
- **切片三**：Twitter 沙盒的 agent 工作流设计——明确它是独立学习环境，不复制公众号逻辑。
- **切片四**：per-category 的"决策 rubric + 信任账本"模板（对应 §3.4 的逐步放行机制）。

**所有后续切片都必须等前一切片验收通过后再启动。这是本项目"复杂性由证据触发"原则的具体落实。**

## 6. 给执行 Agent 的行为约束（防止它跑偏）

明天接手的是一个强模型，它最大的失败模式是**过度工程化**——把简单任务做成漂亮的架构。以下约束是为了把这种倾向压住：

1. **交付可运行的东西，不是设计文档。** 交付物必须是人类今天就能用、能看到真实产出的，不是模块设计或待实现接口。
2. **一次只认领一个任务。** 不要一次铺开做多个切片。
3. **每次产出给选项，不给单一答案。** 强制 3 个候选 + 推荐理由，让人类做选择题，降低人类的判断成本但保留判断回路。
4. **失败行为要明确。** 超时、来源失败、token 超额——必须保留部分结果并明确记录，不能静默崩溃。
5. **自我迭代不是目标，是结果。** 在第一条产出链路稳定跑通多轮之前，不要谈"自我迭代""全自动""Agent 自审批"。

## 7. 待人类决策的开放问题

以下问题本次讨论未最终定论，留待人类进一步思考或与执行 Agent 共同决定：

1. **公众号的领域收窄**：AI 是个很大的领域。是做"AI 通用科普"、"AI 应用实战"、还是某个更细的切面（如"用 AI Agent 做内容/交易"）？定位越窄，方法论越聚焦，也越好做心智占有。
2. **朋友圈触达机制**：§2.2 指出"他们真的读到了内容"是关键之一。公众号文章如何触达朋友圈——转发？群发？还是依赖朋友圈自然曝光？这影响发布节奏的设计。
3. **能力展示的"证据"形态**：除了文章本身，是否要在内容里**显式**展示"这是 Agent 协助产出"的过程（如附调研笔记、候选对比），以强化"AI 应用能力"的定位？还是保持内容干净、把过程留在 self-evo 仓库里作为幕后证据？

## 8. 总结：这次讨论改变了什么

| 维度 | 讨论前 | 讨论后 |
|------|--------|--------|
| 给强模型的指令 | "重构 self-evo" | "跑通最小内容生产闭环" |
| 默认目标 | 做高流量公众号 | 做能力展示，流量非目标 |
| 公众号角色 | 流量号 | 品牌锚点（给朋友圈看） |
| Twitter 角色 | 第二个战场（不建议做） | 学习沙盒（天然的） |
| Agent 与判断 | 倾向让 Agent 替代判断 | Agent 给选项，人做最后一票 |
| 反馈学习 | 直接从自己号学 | 先从别人号学，自己号有权重后再学 |
| 第一步 | 框架/架构设计 | 调研产出"方法论 v1" |

这份文档本身已经是 self-evo 项目的一个交付物——它是"方向校准"类记忆，应当像其它 memory 一样被长期保留。
