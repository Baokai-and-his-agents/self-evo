## 主要发现

1. 保持 Markdown/YAML 和 GitHub 作为当前规模的权威协调层。添加数据库作为派生索引或运行时状态，而非可检查真实来源文件的替代品。
2. 在添加多 agent 编排前测量单 agent 基线。仅为已证明的吞吐量瓶颈引入并行 workers。
3. 在采用外部可观测性平台或持久工作流引擎前构建本地结构化遥测和硬预算/终止控制。
4. **治理边界**：Agent 不得修改 `scripts/` 下的 hooks/policy/validators。Scout runner 路径 `scripts/workers/scout_runner.py` 需先获得权限 proposal 批准。
5. **能力分层**：`rules/RESOURCE_APPROVALS.yaml` 定义批准的能力 scope（如 `public-web-read`），具体来源 allowlist 在 `data/exploration/scout-sources.yaml` 等 registry 实例中定义。登录/token/付费访问需新审批。
6. **Runtime 状态隔离**：cursor、完整 ledger、telemetry、caches 保持 gitignored。仅提交精简 decisions、摘要、schemas。测试验证 gitignore 规则。
4. 复用分布式系统原语：租约、心跳、隔离 worktrees、幂等键、边界重试和人工批准门。
5. 将基准测试百分比、成本估算和升级阈值视为来源特定或临时的，直到 self-evo 有本地测量。
6. OpenViking 是活跃的用户部署。本次运行未解决其公共 repository 身份或基准测试证据，因此未分类为死亡。检查的 GBrain URL 不可用同样不是项目被放弃的证据。
